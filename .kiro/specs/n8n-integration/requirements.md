# Requirements Document

## Introduction

This feature covers the n8n integration layer that bridges Salesforce LWC/Apex with the Claude Vision AI service for form template processing. Two n8n workflows are required:

1. **Template Scan Workflow** — Accepts a form template file (PDF or image) from Salesforce, invokes Claude Vision to extract a structured JSON schema per the agreed contract, validates the schema, and posts the result back to Salesforce asynchronously via a callback REST endpoint.

2. **AI Prompt Edit Workflow** — Accepts a current form schema and a natural-language edit prompt from Salesforce, invokes Claude to apply the requested modifications, validates the output, and returns the updated schema synchronously.

Supporting infrastructure includes API key authentication, a Salesforce Connected App for n8n-to-Salesforce callbacks, Named Credential / Custom Metadata configuration, and Claude prompt engineering to ensure consistent, contract-compliant JSON output.

---

## Glossary

- **n8n_Webhook**: The HTTP endpoint exposed by the n8n instance at `https://n3n3n.suplo.my.id` that receives requests from Salesforce.
- **Scan_Workflow**: The n8n workflow that processes template scan requests asynchronously (Workflow 1).
- **Edit_Workflow**: The n8n workflow that processes AI prompt-based schema edits synchronously (Workflow 2).
- **Claude_Vision**: The Anthropic Claude API invoked by n8n to extract or modify form schemas using vision and language capabilities.
- **Schema_Contract**: The agreed JSON schema format with field types: `text`, `textarea`, `number`, `date`, `time`, `checkbox`, `picklist`, `table`, `signature`; structure: `section → row → field`; 12-column grid (`colSpan`).
- **Callback_Resource**: The Salesforce Apex REST endpoint `/services/apexrest/formschema/v1/callback` that receives async results from n8n.
- **Connected_App**: The Salesforce OAuth 2.0 Connected App using client credentials flow, used by n8n to authenticate callbacks to Salesforce.
- **Named_Credential**: The Salesforce Named Credential `n8n_Form` storing the n8n base URL (`https://n3n3n.suplo.my.id`) and auth header configuration.
- **External_Credential**: The Salesforce External Credential holding the API key principal for n8n authentication.
- **Custom_Metadata**: Salesforce Custom Metadata records storing non-secret n8n configuration such as webhook paths and Claude tuning parameters.
- **FormSchemaValidator**: The existing Apex class that validates schema invariants (id uniqueness, colSpan ≤ 12, type whitelist, signature/picklist/table rules).
- **n8n_Schema_Validator**: The validation logic implemented inside n8n workflows that mirrors FormSchemaValidator rules for early rejection before callback.
- **jobId**: A UUID generated by Salesforce (FormScanQueueable) uniquely identifying an async scan job.
- **templateId**: The Salesforce record ID of the Form_Template__c being scanned or edited.
- **schemaVersion**: The integer version of the Schema_Contract format (not the template content version).
- **Integration_User**: A dedicated Salesforce user account with minimal permissions used exclusively by n8n for callback authentication.

## Requirements

### Requirement 1: Template Scan Webhook Authentication

**User Story:** As a Salesforce administrator, I want all requests to the n8n template scan webhook to be authenticated, so that unauthorized callers cannot trigger scan jobs or consume AI quota.

#### Acceptance Criteria

1. THE Scan_Workflow SHALL expose a webhook at path `/webhook/template-scan` on the n8n instance.
2. WHEN a request is received at `/webhook/template-scan`, THE Scan_Workflow SHALL verify that the `X-API-Key` HTTP header is present and that its value matches the configured API key secret, using case-insensitive comparison for the header name.
3. IF the `X-API-Key` header is absent or its value does not match the configured API key secret, THEN THE Scan_Workflow SHALL return HTTP 401 and SHALL NOT execute any downstream workflow nodes, terminating processing at the authentication step.
4. THE Named_Credential `n8n_Form` SHALL store the header name `X-API-Key` so that Apex callouts include the correct header on every request.
5. THE External_Credential principal SHALL store the API key value, and the Scan_Workflow SHALL retrieve the key exclusively from the External_Credential principal, never from Custom Metadata or Apex source code.

---

### Requirement 2: Template Scan Request Acceptance

**User Story:** As the FormScanQueueable Apex class, I want the scan webhook to acknowledge receipt immediately, so that the Salesforce Queueable job does not time out waiting for a synchronous response.

#### Acceptance Criteria

1. WHEN a valid authenticated POST request is received at `/webhook/template-scan`, THE Scan_Workflow SHALL return HTTP 200 with body `{"status": "accepted", "jobId": "<jobId>"}` within 5 seconds.
2. WHEN the HTTP 200 acknowledgement is sent, THE Scan_Workflow SHALL continue processing the scan job asynchronously without blocking the HTTP response.
3. THE Scan_Workflow SHALL accept a JSON request body containing: `templateId` (string), `fileContentVersionId` (string), `fileMimeType` (string), `fileBase64` (string), `callbackUrl` (string), `jobId` (string UUID).
4. IF any of the required fields `templateId`, `fileBase64`, `fileMimeType`, `callbackUrl`, `fileContentVersionId`, or `jobId` are missing from the request body, THEN THE Scan_Workflow SHALL return HTTP 400 with a descriptive error message, SHALL NOT return HTTP 200, and SHALL NOT invoke Claude_Vision.
5. IF `fileMimeType` is present but its value is not `application/pdf` or a value matching the pattern `image/*`, THEN THE Scan_Workflow SHALL return HTTP 400 with a descriptive error message and SHALL NOT invoke Claude_Vision.
6. IF `jobId` is present but does not conform to UUID format (8-4-4-4-12 hexadecimal characters), THEN THE Scan_Workflow SHALL return HTTP 400 with a descriptive error message and SHALL NOT invoke Claude_Vision.

---

### Requirement 3: Claude Vision Schema Extraction

**User Story:** As a form builder user, I want Claude Vision to extract a structured JSON schema from an uploaded template file, so that I do not have to manually recreate the form structure in the editor.

#### Acceptance Criteria

1. WHEN the Scan_Workflow processes a scan job, THE Scan_Workflow SHALL invoke Claude_Vision with the template file using a PDF native document block when `fileMimeType` is `application/pdf`, and an image base64 block for all other MIME types.
2. THE Scan_Workflow SHALL instruct Claude_Vision via a system prompt to output only a valid JSON object conforming to the Schema_Contract, with no preamble, markdown fences, explanation text, or trailing content outside the JSON.
3. THE Scan_Workflow SHALL instruct Claude_Vision to structure the extracted schema as: a root object with `schemaVersion` (integer, value `1`), `sections` (array), where each section contains `id` (string, unique form-wide), `label` (string), and `rows` (array), where each row contains `fields` (array), where each field contains at minimum `id` (string, unique form-wide), `type` (one of `text`, `textarea`, `number`, `date`, `time`, `checkbox`, `picklist`, `table`, `signature`), `label` (string), and `colSpan` (integer 1–12).
4. THE Scan_Workflow SHALL instruct Claude_Vision that the sum of `colSpan` values across all fields within a single row SHALL NOT exceed 12, and each row SHALL contain at least one field with `colSpan` ≥ 1.
5. THE Scan_Workflow SHALL instruct Claude_Vision that fields of type `signature` require a `signerRole` property with value `engineer` or `supervisor`.
6. THE Scan_Workflow SHALL instruct Claude_Vision that fields of type `picklist` require an `options` property containing a non-empty array of string values.
7. THE Scan_Workflow SHALL instruct Claude_Vision that fields of type `table` require a `columns` property containing a non-empty array of column descriptor objects, where each column descriptor contains at minimum `id` (string) and `label` (string).
8. THE Scan_Workflow SHALL instruct Claude_Vision that the field type `html` is forbidden (anti-XSS constraint).
9. IF Claude_Vision returns a non-2xx HTTP response or the response body cannot be parsed as a JSON object, THEN THE Scan_Workflow SHALL treat the invocation as a failure and proceed to post an error callback to the `callbackUrl` with `status: "error"` and an error description identifying the Claude invocation failure.

### Requirement 4: Scan Output Validation in n8n

**User Story:** As a system integrator, I want the n8n scan workflow to validate Claude's output before sending it back to Salesforce, so that only schema-contract-compliant JSON reaches the Callback_Resource.

#### Acceptance Criteria

1. WHEN Claude_Vision returns a response, THE n8n_Schema_Validator SHALL attempt to parse the response as JSON; IF parsing fails, THE n8n_Schema_Validator SHALL treat the response as a validation failure and proceed to the error callback path defined in criterion 8.
2. THE n8n_Schema_Validator SHALL verify that every field `id` is non-empty, a string, and unique across the entire extracted schema.
3. THE n8n_Schema_Validator SHALL verify that the sum of `colSpan` values within each row does not exceed 12; fields with a missing `colSpan` property SHALL be treated as having a `colSpan` value of `0` for this calculation.
4. THE n8n_Schema_Validator SHALL verify that every field `type` value belongs to the whitelist: `text`, `textarea`, `number`, `date`, `time`, `checkbox`, `picklist`, `table`, `signature`.
5. THE n8n_Schema_Validator SHALL verify that `signature` fields contain a `signerRole` property equal to `engineer` or `supervisor`.
6. THE n8n_Schema_Validator SHALL verify that `picklist` fields contain a non-empty `options` array.
7. THE n8n_Schema_Validator SHALL verify that `table` fields contain a non-empty `columns` array.
8. IF the n8n_Schema_Validator detects a validation failure, THEN THE Scan_Workflow SHALL set `status` to `"error"` and SHALL include in the callback payload an `error` string that identifies the failed validation rule and the `id` of the offending field, or indicates JSON parse failure if parsing was the cause.

### Requirement 5: Asynchronous Scan Callback to Salesforce

**User Story:** As the FormCallbackResource Apex REST endpoint, I want n8n to POST the scan result to the callbackUrl provided in the original request, so that Salesforce can update the Form_Template__c record and notify the user.

#### Acceptance Criteria

1. WHEN scan processing completes (success or validation error), THE Scan_Workflow SHALL POST to the `callbackUrl` received in the original request body.
2. WHEN the Scan_Workflow is ready to POST the callback, THE Scan_Workflow SHALL authenticate the callback POST to Salesforce using OAuth 2.0 client credentials flow via the Connected_App, obtaining a bearer token before the POST.
3. THE callback POST body SHALL contain: `jobId` (string, matching the original request), `templateId` (string), `status` (string, exactly `"success"` or `"error"`), `schema` (object, present and non-null when status is `"success"`, omitted when status is `"error"`), `error` (string, present and non-null when status is `"error"`, omitted when status is `"success"`). Each POST request SHALL time out after 30 seconds if no response is received.
4. WHEN the callback POST receives HTTP 2xx from Salesforce, THE Scan_Workflow SHALL log the successful delivery and mark the job as completed.
5. IF the callback POST receives a non-2xx response, times out, or encounters a network error, THEN THE Scan_Workflow SHALL retry the callback up to 3 times using exponential backoff (initial delay 5 seconds, multiplier 2, maximum 3 attempts). After all retry attempts are exhausted, THE Scan_Workflow SHALL mark the job as permanently failed and SHALL NOT attempt further retries.
6. THE Scan_Workflow SHALL log the final callback HTTP status and each retry attempt number for observability.
7. IF the OAuth token request fails, THEN THE Scan_Workflow SHALL retry token acquisition up to 3 times; after all token acquisition attempts are exhausted, THE Scan_Workflow SHALL mark the callback as permanently failed, log the OAuth failure, and SHALL NOT attempt the callback POST.
8. THE Scan_Workflow SHALL record the total elapsed time from webhook receipt to callback completion in milliseconds in its final log entry.

### Requirement 6: AI Prompt Edit Webhook Authentication and Request Contract

**User Story:** As a form builder user, I want to send a natural-language prompt to modify an existing schema and receive the updated schema synchronously, so that I can iteratively refine my form template without re-scanning.

#### Acceptance Criteria

1. THE Edit_Workflow SHALL expose a webhook at path `/webhook/schema-edit` on the n8n instance.
2. WHEN a request is received at `/webhook/schema-edit`, THE Edit_Workflow SHALL verify the `X-API-Key` header using the same secret as the Scan_Workflow, using case-insensitive comparison for the header name.
3. IF the `X-API-Key` header is absent or its value is invalid, THEN THE Edit_Workflow SHALL return HTTP 401, SHALL NOT return any 2xx response, and SHALL NOT invoke Claude_Vision.
4. THE Edit_Workflow SHALL accept a JSON request body containing: `templateId` (string, non-empty, maximum 255 characters), `currentSchema` (object conforming to Schema_Contract, non-null), `prompt` (string, non-empty after trimming whitespace, maximum 2000 characters).
5. IF the request body is absent, is not valid JSON, or cannot be parsed as a JSON object, THEN THE Edit_Workflow SHALL return HTTP 400 with an error message indicating a body parsing failure.
6. IF any of the required fields `templateId`, `currentSchema`, or `prompt` are missing, empty, whitespace-only (for `prompt`), or if `currentSchema` is not a valid Schema_Contract object, THEN THE Edit_Workflow SHALL return HTTP 400 with an error message indicating which field failed validation.

---

### Requirement 7: Claude AI Schema Modification

**User Story:** As a form builder user, I want Claude to apply my natural-language edit prompt to the current schema and return a modified schema, so that structural changes are fast and do not require re-uploading the template file.

#### Acceptance Criteria

1. WHEN the Edit_Workflow receives a request where `currentSchema` is a non-empty object and `prompt` is a non-empty string, THE Edit_Workflow SHALL invoke Claude_Vision with a system prompt containing the full `currentSchema` serialized as JSON and the user-supplied `prompt`.
2. THE Edit_Workflow SHALL instruct Claude_Vision to return only a valid JSON object with fields: `schema` (the modified Schema_Contract object) and `changes` (array of strings where each string names the field modified and the type of change applied).
3. THE Edit_Workflow SHALL instruct Claude_Vision that field `id` values are immutable: renaming or deleting existing field ids is forbidden.
4. THE Edit_Workflow SHALL instruct Claude_Vision that the sum of `colSpan` values within each row SHALL NOT exceed 12 in the output schema.
5. THE Edit_Workflow SHALL instruct Claude_Vision that `html` is a forbidden field type.
6. THE Edit_Workflow SHALL instruct Claude_Vision to output no preamble, markdown fences, explanation text, or trailing content outside the JSON object.
7. IF Claude_Vision returns a non-2xx HTTP response or a network error, THEN THE Edit_Workflow SHALL treat the invocation as a failure and proceed to the retry defined in Requirement 8 criterion 3.
8. WHEN Claude_Vision returns a 2xx response, THE Edit_Workflow SHALL parse the response body as JSON; IF parsing fails or the parsed value is not an object containing both `schema` and `changes` keys, THEN THE Edit_Workflow SHALL treat the response as a validation failure and proceed to the retry defined in Requirement 8 criterion 3.

---

### Requirement 8: Edit Output Validation and Retry

**User Story:** As a system integrator, I want the edit workflow to validate Claude's output and retry once on failure, so that transient formatting errors from Claude do not propagate invalid schemas to Salesforce.

#### Acceptance Criteria

1. WHEN Claude_Vision returns a parseable response for an edit request, THE n8n_Schema_Validator SHALL apply the same structural validation rules as defined in Requirement 4 (id uniqueness, colSpan ≤ 12, type whitelist, signature/picklist/table rules).
2. THE n8n_Schema_Validator SHALL additionally verify that no field `id` value present in `currentSchema` is absent from the output schema; a field `id` from `currentSchema` that does not appear in the output schema constitutes a removal violation regardless of any new ids added by the edit.
3. IF the n8n_Schema_Validator detects a validation failure on the first attempt, OR IF Claude_Vision returns a non-2xx HTTP response or network error on the first attempt, THEN THE Edit_Workflow SHALL invoke Claude_Vision exactly once more with the original `currentSchema`, the original `prompt`, and an additional instruction describing the specific validation failure or invocation error from the first attempt.
4. IF the n8n_Schema_Validator detects a validation failure on the second attempt, OR IF the retry invocation fails due to a network error or service unavailability, THEN THE Edit_Workflow SHALL return HTTP 422 with `{"status": "error", "error": "<validation failure or invocation failure description>"}` and SHALL NOT return a partial schema.
5. WHEN the edited schema passes validation, THE Edit_Workflow SHALL return HTTP 200 with `{"status": "success", "schema": <modified schema object>, "changes": <array of one or more human-readable change description strings>}`, and SHALL NOT return HTTP 200 when schema validation has failed.
6. THE Edit_Workflow SHALL complete and return a response within 60 seconds of receiving the request.

### Requirement 9: Salesforce Connected App for n8n Callback Authentication

**User Story:** As a Salesforce administrator, I want n8n to authenticate to Salesforce using a dedicated Connected App and integration user, so that callback permissions are minimal and auditable.

#### Acceptance Criteria

1. THE Connected_App SHALL be configured with OAuth 2.0 client credentials flow enabled in its settings.
2. THE Integration_User SHALL be assigned only the following permissions: HTTP POST access to the Callback_Resource endpoint (`/services/apexrest/formschema/v1/callback`), Read access on `Form_Template__c.Status__c`, and Edit access on `Form_Template__c.Schema_JSON__c` and `Form_Template__c.Status__c`.
3. THE Integration_User SHALL NOT be assigned the Form_Builder_User permission set, a System Administrator profile, or any profile with "Modify All Data" or "View All Data" permissions.
4. WHEN the Scan_Workflow is about to POST a callback, THE Scan_Workflow SHALL obtain an OAuth 2.0 access token from the Salesforce token endpoint using the Connected_App client ID and secret stored as n8n credentials (not in Custom Metadata or Apex source).
5. IF the OAuth token request fails, THEN THE Scan_Workflow SHALL retry token acquisition up to 3 attempts; WHEN all 3 token acquisition attempts have failed, THE Scan_Workflow SHALL mark the callback as permanently failed, log the final OAuth error response to the n8n workflow execution log, and SHALL NOT attempt the callback POST.
6. IF a token acquisition attempt succeeds after the 3-attempt retry limit has already been exceeded and the callback has already been marked as permanently failed, THE Scan_Workflow SHALL still treat the callback as permanently failed and SHALL NOT use the late-acquired token.

### Requirement 10: Custom Metadata and Named Credential Configuration

**User Story:** As a Salesforce developer, I want non-secret n8n configuration stored in Custom Metadata and credentials stored in Named/External Credentials, so that configuration is deployable via SFDX and secrets are never committed to source control.

#### Acceptance Criteria

1. THE Custom_Metadata SHALL store the following fields: `scan_webhook_path` (string, default `/webhook/template-scan`), `edit_webhook_path` (string, default `/webhook/schema-edit`), `claude_model_id` (string), `callback_max_retries` (integer, 1–10), `claude_temperature` (decimal, 0.0–1.0), and `claude_max_tokens` (integer, 1–4096).
2. THE Named_Credential `n8n_Form` SHALL store the base URL `https://n3n3n.suplo.my.id` and the authentication header name `X-API-Key`.
3. THE External_Credential principal SHALL store the API key value and SHALL be referenced by the Named_Credential via a principal assignment; correct linkage is confirmed when callouts from Apex succeed and no API key value appears in Apex source or Custom_Metadata records.
4. THE Custom_Metadata SHALL NOT store API key values, OAuth access tokens, OAuth client secrets, passwords, or private key material.
5. WHEN the Named_Credential base URL or auth header name changes, the callout configuration in Apex SHALL reference only the Named_Credential record, and no hardcoded base URL or auth header name SHALL appear in Apex source code.
6. WHEN the Custom_Metadata and Named_Credential records are deployed via SFDX to a target org, the deployment SHALL complete without errors and all six Custom_Metadata fields defined in criterion 1 SHALL be present and readable in the target org.

---

### Requirement 11: Schema Round-Trip Integrity

**User Story:** As a form builder user, I want confidence that schemas extracted by Claude Vision and subsequently edited via AI prompt maintain their structural integrity, so that the rendered form always matches the original template layout intent.

#### Acceptance Criteria

1. WHEN the Scan_Workflow outputs a schema in a `"success"` callback, applying the n8n_Schema_Validator to that schema SHALL produce no validation errors.
2. WHEN the Edit_Workflow returns a `"success"` response, the output `schema` SHALL contain all field `id` values present in the input `currentSchema`; new field `id` values MAY be added by the edit, but no existing `id` SHALL be absent. IF any `id` from `currentSchema` is absent in the output, THE Edit_Workflow SHALL return HTTP 422 with `{"status": "error", "error": "<id preservation violation description>"}`.
3. WHEN the Scan_Workflow or Edit_Workflow outputs a schema, for every row in that schema the sum of `colSpan` values of all fields in that row SHALL be greater than 0 and SHALL NOT exceed 12, and each row SHALL contain at least one field with `colSpan` ≥ 1; IF this constraint is violated, the workflow SHALL treat it as a validation failure and proceed to the error path.
4. WHEN the Scan_Workflow posts a `"success"` callback to Salesforce, the `schema` field in the callback body SHALL be parseable by `FormSchemaValidator.validate()` without errors.
5. WHEN the Edit_Workflow returns a `"success"` response, the `schema` field SHALL be parseable by `FormSchemaValidator.validateEdit(currentSchema, outputSchema)` without errors, where `currentSchema` is the object received in the original request and `outputSchema` is the schema returned in the response. `validateEdit()` checks include: all invariants from `validate()`, plus the id preservation rule (no existing id removed), plus the no-rename constraint.

### Requirement 12: Observability and Error Reporting

**User Story:** As a system administrator, I want meaningful logs and error payloads from both n8n workflows, so that I can diagnose scan and edit failures without access to the n8n execution UI.

#### Acceptance Criteria

1. THE Scan_Workflow SHALL include the `jobId` and `templateId` in every log entry produced during processing of a scan job at INFO severity or higher; IF `jobId` or `templateId` is unavailable at the time of logging, THE Scan_Workflow SHALL use the literal string `"unknown"` as the placeholder value.
2. THE Edit_Workflow SHALL include the `templateId` and the first 100 characters of `prompt` in every log entry produced during processing of an edit request at INFO severity or higher; IF `templateId` is unavailable, THE Edit_Workflow SHALL use the literal string `"unknown"` as the placeholder value.
3. IF Claude_Vision returns a non-2xx HTTP response, THEN THE Scan_Workflow or Edit_Workflow (as applicable) SHALL log the HTTP status code and the first 500 characters of the response body.
4. IF the n8n_Schema_Validator rejects a Claude_Vision response, THEN THE Scan_Workflow or Edit_Workflow SHALL log the specific validation rule(s) that failed and the dot-notation path(s) to the offending field(s) within the schema object (for example, `sections[0].rows[1].fields[2].colSpan`).
5. THE Scan_Workflow SHALL record the total elapsed time in milliseconds from webhook receipt to receipt of the HTTP response to the outbound callback POST in its final log entry.
6. THE Edit_Workflow SHALL record the total elapsed time from webhook receipt to HTTP response in milliseconds in its final log entry.

### Requirement 13: Test Template Coverage for Claude Prompt Validation

**User Story:** As a QA engineer, I want a defined set of test templates that exercise edge cases in Claude Vision extraction, so that prompt engineering is validated across representative form complexity levels before production use.

#### Acceptance Criteria

1. THE test suite SHALL include a simple flat form template, distinct from all other test templates, containing only `text`, `number`, and `date` fields with no tables, signatures, or multi-section layout.
2. THE test suite SHALL include a multi-section form template, distinct from all other test templates, with at least 3 sections and mixed field types including `picklist` and `checkbox`.
3. THE test suite SHALL include a form template, distinct from all other test templates, containing at least one `table` field with at least 2 columns.
4. THE test suite SHALL include a form template, distinct from all other test templates, containing exactly two `signature` fields with `signerRole` values `engineer` and `supervisor` respectively.
5. THE test suite SHALL include a form template, distinct from all other test templates, with a row that uses a 12-column layout (e.g., one field with `colSpan: 12` or multiple fields whose `colSpan` values sum to exactly 12) to verify column constraint enforcement.
6. WHEN each test template is submitted to the Scan_Workflow for the first time without any manual schema correction between execution and validation, THE Scan_Workflow SHALL post a callback with `status: "success"` AND the n8n_Schema_Validator SHALL report zero validation errors for the returned schema.
7. WHEN each test template produces a `status: "success"` callback, THE Callback_Resource SHALL return HTTP 2xx AND the corresponding `Form_Template__c` record's `Status__c` field SHALL be updated to reflect the successful scan result.
